## ⚠️ 세션(Session) 방식 로그인의 한계

웹툰은 콘텐츠 소비이므로 <sup>1)</sup>사용자가 많고 <sup>2)</sup>사용자가 모바일/PC/앱 등 다양한 환경에서 접속합니다.
그래서 인증 방식은 아래 두 요구사항을 만족해야 했습니다.

### 요구사항 1) Scale-out 환경에서 로그인 상태가 안정적으로 유지되어야 함
트래픽이 커지면서 서버의 수평 확장(Scale-out)을 생각하게 되었지만 세션 방식을 사용시 세션을 서버 메모리(in-memory) 에 저장하면, 사용자가 '서버1'에서 로그인한 뒤 로드밸런싱으로 '서버2'로 라우팅될 경우 세션을 찾지 못해 로그인이 풀릴 수 있습니다.
- 대안: 중앙 세션 저장소 사용(예: Redis)
  - 세션 기반 인증은 요청마다 세션 조회가 발생하므로 트래픽이 커질수록 Redis 조회 QPS가 증가합니다.
  - 포트폴리오 운영 관점에서는 비용/지연/장애 대응 부담이 커져 세션 방식은 채택하지 않았습니다.

### 요구사항 2) 모바일 웹/앱 환경에서도 인증이 일관되어야 함
웹툰은 모바일 웹 & 앱을 통해 소비하는 경우가 가장 두드러지고 세션 방식을 사용시 앱에서 쿠키 정책, 저장 환경이 환경별로 불안정해 로그인 유지가 깨질 수 있습니다.

---

## 🛠️ 해결방법

서비스는 트래픽 증가에 따라 수평 확장(Scale-out) 이 필요했고, 서버가 인증 상태(세션)를 들고 있으면 확장 시 구조가 복잡해집니다.
따라서 인증은 서버 상태에 의존하지 않는 Stateless 방식이 적합하다고 판단했고, 최종적으로 JWT 기반 토큰 인증(Access/Refresh) 을 채택했습니다.

## 📌 Refresh Token의 저장 방식 이슈

Creatorhub 서비스의 Access Token의 유효시간은 15분입니다. 만료시 토큰을 갱신할 방법으로 Refresh Token을 사용해야 했고, 
<sup>1)</sup>Refresh Token을 어디에 저장하고 <sup>2)</sup>저장한다면 저장소는 무엇을 해야하는지에 대한 이슈가 있었습니다.

### 이슈 1) 저장방식은? Cookie vs 별도의 저장소
처음엔 HttpOnly, Secure, Samesite 등을 고려한 쿠키에 저장하는 방식을 생각했으나 보안상 '강제 로그아웃' 기능 적용 및 토큰 탈취 확률을 줄이고 싶었습니다.

### 이슈 2) 저장소는? RDB vs Redis
운영을 고려한다면 RDB 였지만 사용하지 않게된 Refresh Token을 Refresh Token의 유효기간에 맞춰 제거하는 것이 중요하다고 생각되었고
운영 비용을 고려한다면 RDB 였고 배치로 만료된 토큰의 제거는 가능하지만, Refresh Token 검증은 
요청 경로의 핫패스라 지연에 민감하고, 
만료 데이터 정리를 위한 배치 운영 부담이 생긴다. Redis는 TTL로 자동 정리되며, 
refresh 세션 검증을 빠르게 처리할 수 있어 저장소로 선택했다.
TTL기능이 있는 Redis를 저장소로 선정했습니다.

## 🔥 최종 결론 및 요약
서버의 수평적 확장을 고려해 Serverless 방식인 JWT 토큰을 적용했으며
Refresh Token의 저장 방식 문제는 토큰 탈취 확률과 강제 로그아웃 가능 등 보안상 문제를 줄이기 위해
Redis를 저장소로 택했습니다. Redis는 TTL 서비스를 위해 저장소로 선정했습니다.
